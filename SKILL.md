# DEST Core Buffer Skill（core/）工程级强制流程版（完全解耦）

## 0) 目标
把 core/ 中“缓冲 + 重排 + 通道拆分”问题落到**可证明正确性、可量化性能、可最小改动回归**的工程方案中。  
**只允许改 core/**；外部模块只通过“接口契约”交互。

输出必须包含：对外契约确认、不变量列表、热路径成本（拷贝/分配/锁）、最小改动点（文件+函数）、验证矩阵（满/空/回卷/乱序/重复/缺失）、性能与验收标准、风险与回滚点。

---

## 1) 启用后的最小澄清清单（信息不足时最多只问这些）
最多追问 6 类：
1) **数据单位**：输入是 packet / frame / samples 里的哪一种？（必须选一个）  
2) **顺序与缺失**：可能乱序/重复/缺失吗？seq/ts/offset 哪个可用？  
3) **线程模型**：SPSC / MPSC / MPMC？（决定锁/无锁结构）  
4) **所有权**：输入内存生命周期是“调用期有效”还是“可长期持有”？能否零拷贝？  
5) **输出语义**：严格单调 / 尽力有序 / 允许缺口标记？缺口如何表达（补零/丢弃/标记）？  
6) **规模指标**：吞吐（MB/s 或 samples/s）、最大乱序窗口、可接受延迟（P95/P99）？

> 用户已给出足够信息时，禁止重复追问。

## 3) 强制工程流程（必须按 Step 输出，不可跳步）

### Step 1：对外接口契约确认（必须先确认后改动）
#### 3.1 输入契约（来自外部调用者）
必须明确并写出结论：
- 输入数据单元：**packet / frame / samples**（必须选一个并说明依据）
- 输入顺序保证：已排序 / 可能乱序 / 可能重复 / 可能缺失（必须说明）
- 输入线程模型：SPSC / MPSC / MPMC（必须说明）
- 输入内存所有权（二选一必须定）：
  - A. 仅调用期间有效（本模块需拷贝/引用计数）
  - B. 可长期持有（可零拷贝持有）

> 只要“所有权/生命周期”不清楚，禁止做“减拷贝/零拷贝”优化（会引入悬垂引用）。

#### 3.2 输出契约（交付给外部调用者）
必须明确并写出结论：
- 输出单元：frame / window / chunk
- 输出顺序语义：严格单调 / 尽力有序 / 允许缺口标记
- 缺口表达：丢弃 / 补零 / 标记缺口 / 返回错误（必须选）
- 输出生命周期：返回对象是否复用内部缓冲、是否线程安全（必须明确）

---

### Step 2：模块内不变量清单（改动必须保持）
必须列出并逐条对照（至少包含以下）：

#### 2.1 RingBuffer 不变量
- 容量单位：capacity 以 **输入/输出单元**计（必须写清）
- 指针关系：`0 ≤ used ≤ capacity`
- 回卷规则：read/write 指针回卷条件与处理方式
- 满策略：覆盖 / 丢弃 / 阻塞 / 回压（**只能选一种并保持一致**）
- 可见性：并发下读写指针更新的原子性/内存序（如适用）

#### 2.2 ReorderBuffer 不变量
- 乱序窗口大小定义（单位与上限）
- 去重规则：基于 seq / ts / offset 哪一个（必须明确）
- 输出规则：如何保证输出语义（严格单调或缺口标记）
- 缺失处理：缺口何时确认、如何推进输出

#### 2.3 ChannelSeparator 不变量
- 通道对齐定义：按样本/按帧/按时间戳（必须明确）
- 拆分后长度与对应关系可证明（例如每通道样本数一致/差异规则明确）
- `uint24` 解包/对齐规则与端序假设（如相关）

---

### Step 3：现状热路径成本盘点（必须给“清单 + 证据”）
必须逐条给结论：
- **拷贝点清单**：发生几次 memcpy？每次大小？在哪些函数？
- **分配点清单**：哪些函数会 new/delete 或触发容器扩容？频率？
- **锁开销**：锁在哪里？临界区做了什么？是否可缩小？
- **长跑稳定性**：是否存在周期性 latency spike（扩容/碎片/锁竞争/缓存抖动）？

---

### Step 4：正确性问题归因（必须分型）
必须把问题归因到至少一种类型并说明证据：
- 溢出/覆盖策略与上层期望不一致（导致“错位/跳变”）
- 回卷边界处理错误（wrap-around off-by-one）
- 乱序窗口不足/过大导致的输出异常（卡住/延迟爆炸）
- 去重规则错误（重复包导致双写/错位）
- 通道拆分对齐错误（步长/交错顺序/端序/uint24 解包）
- 并发可见性问题（读写指针/元数据更新时序）

---

### Step 5：最小改动方案（必须落到文件+函数 + 不变量对照）
每条改动必须包含：
- 文件名 + 函数名
- 修改内容（可实现的动作）
- 预期收益（正确性/吞吐/延迟/内存）
- 不变量如何保持（引用 Step 2 的条目逐条说明）
- 风险与回归项（仅限本模块）

**优先级建议**（从高到低）：
1) 修正不变量/边界（回卷、满策略、缺口推进）  
2) 减少热路径分配（预分配、reserve、对象池）  
3) 减少拷贝（批量拷贝/视图/引用计数块——前提：所有权明确）  
4) 锁优化（缩小临界区；满足 SPSC 时可替换为无锁环）  
5) 重排结构优化（避免“全排序”热路径；用窗口桶/索引表/位图）

---

### Step 6：验证矩阵（必须覆盖 + 可执行）
必须输出“验证用例矩阵”，至少包含：

#### 6.1 RingBuffer
- 空/满边界
- 覆盖/丢弃策略一致性
- 回卷读写（跨边界写、跨边界读）
- 连续写读与随机分段写读
- 并发场景（按 SPSC/MPSC/MPMC 约束设计）

#### 6.2 ReorderBuffer
- 乱序输入（窗口内/窗口外）
- 重复输入（同 seq/ts）
- 缺失输入（缺口标记/补零/丢弃策略）
- 延迟推进（缺口确认策略是否导致卡死）
- 峰值乱序压力（窗口上限下的稳定性）

#### 6.3 ChannelSeparator / uint24
- 单通道/多通道
- 交错数据拆分正确性（对齐、长度一致）
- uint24 解包正确性（端序、符号扩展、范围）
- 性能压力（大块批处理 vs 逐样本）

---

### Step 7：性能与验收标准（必须量化）
必须给出指标与阈值建议（按项目填）：
- 吞吐：MB/s 或 samples/s（稳态 + 峰值）
- 延迟：P50/P95/P99（尤其是 reorder 输出到达延迟）
- 拷贝量：每输入单元的 memcpy 次数/字节数
- 分配次数：每秒分配次数（目标趋近 0）
- 锁竞争：锁等待占比/临界区耗时（如适用）
- 长稳：运行 N 小时无周期性 spike、无内存增长

验收口径必须明确：达到哪些值算“修好”。

---

## 4) 性能规则（模块内）

### 4.1 禁止行为（出现即否决）
- 热路径 new/delete 或隐式扩容（vector/QVector push_back 不 reserve）
- 用粗粒度锁包住整个写/读流程
- 为了重排每次对全缓存排序（sort in hot path）
- 通道拆分逐样本 push（必须批量/视图/预分配）

### 4.2 必做性能检查（回答必须逐条给结论）
- 拷贝点清单（次数/大小/位置）
- 分配点清单（位置/频率）
- 锁临界区（做了什么、如何缩小）
- 长跑稳定性（spike 来源定位）

3) 最小改动方案（按文件+函数列出）

6) 性能与验收标准（量化指标）
